# Plan

Single source of truth for the active roadmap.

## Active

### ⏳ Rich GitHub release notes (gitmap-v19 style)

**Goal:** every published jpg2pdf release page should look like
[gitmap-v19 v5.4.21](https://github.com/alimtvnetwork/gitmap-v19/releases/tag/v5.4.21):
changelog, Release Info table, SHA256 checksums block, multi-OS Quick Install
snippets, pinned-version installer snippets, manual-download note, and an
Assets table — auto-generated by GitHub Actions on every tag push.

Current state: `.github/workflows/release.yml` only sets
`generate_release_notes: true` and uploads binaries. No body template, no
checksums block, no install snippets, no assets table.

Execute in 5 ordered steps. Do ONE step per "next" from the user.

#### Step 1 — Build the release-notes template
- Add `.github/release-notes-template.md` with placeholders: `{{VERSION}}`,
  `{{COMMIT_SHORT}}`, `{{BRANCH}}`, `{{BUILD_DATE_UTC}}`, `{{PYTHON_VERSION}}`,
  `{{CHANGELOG}}`, `{{CHECKSUMS}}`.
- Sections in order: heading + tag link, Changelog, `---`, `## Release Info`
  table, `## Checksums (SHA256)` fenced block, `## Install` (Windows
  PowerShell quick install, Linux/macOS quick install, pinned-version
  installers, Manual download note), `## Assets` platform/arch/file table.

#### Step 2 — Wire CHANGELOG extraction
- Create top-level `CHANGELOG.md` seeded with `## [1.1.0]` entries
  (mixed-input merge, subtle default, hardened `run.ps1`).
- Add a release job step that extracts the section for the current tag
  (between `## [X.Y.Z]` and the next `##`). Falls back to
  `No changelog entry found for {{VERSION}}.` when missing.

#### Step 3 — Generate checksums + render template in workflow
- After all build matrix jobs upload artifacts, in the `release` job:
  - Already runs `sha256sum * > SHA256SUMS.txt` — also capture that output
    into a variable for the template.
  - Compute `COMMIT_SHORT`, `BUILD_DATE_UTC`, `PYTHON_VERSION` from the
    runner / `github` context.
  - `sed`-substitute placeholders in the template → `dist/RELEASE_NOTES.md`.

#### Step 4 — Hand the rendered body to the GitHub Release
- Change `softprops/action-gh-release@v2` step to use `body_path:
  dist/RELEASE_NOTES.md` instead of `generate_release_notes: true`.
- Keep the existing `files:` upload list. Add `install.ps1` and `install.sh`
  to the uploaded files so the "Quick install" URLs in the body resolve.

#### Step 5 — Verify on a throwaway tag
- Push `v1.1.0-rc1` (or re-tag a test tag) and confirm the rendered release
  page matches the gitmap layout: changelog text, Release Info table,
  checksum block, all three install snippets, assets table.
- If anything is missing or mis-rendered, iterate on the template, then
  delete the rc tag and ship `v1.1.0` proper.
